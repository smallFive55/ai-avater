# 更新记忆

将用户输入同步到 `memory/` 下的对应记忆文件，无需用户手写「更新记忆到 memory」等完整指令。

## 记忆类型映射

| 用户说法 / 关键词 | 目标位置 | 文件命名 |
|------------------|----------|----------|
| 决策、decisions | `memory/decisions/` | decision-NNN.md（N 为下一序号） |
| 成功、successes | `memory/successes/` | success-NNN.md |
| 失败、failures | `memory/failures/` | failure-NNN.md |
| 冲突、conflicts | `memory/conflicts/` | conflict-NNN.md |
| 技能栈、skills | `memory/skills/skills.md` | 单文件，追加或更新 |
| 部门重点、focus | `memory/department-focus/` | focus-YYYYMMDD.md |
| 记录完成、完成、已完成 | `memory/completed-tasks/` | completed-YYYYMMDD.md（按当日日期） |

若用户未指定类型，根据内容语义推断；无法推断时，询问用户要记录到哪种记忆。

## 执行流程

1. **解析意图**：从用户输入识别要记录的记忆类型（决策/成功/失败/冲突/技能/部门重点/已完成）。
2. **确定文件**：决策/成功/失败/冲突需新建文件，取下一可用序号；技能更新 skills.md；部门重点用当日日期命名；已完成用当日日期命名 completed-YYYYMMDD.md。
3. **判断来源是否为 Word 文档**：若用户提供或引用的记忆来源是 `.docx` 文件，则按「Word 文档（含图片）处理流程」先提取并总结内容，再进入步骤 4；否则直接进入步骤 4。
4. **读取模板**：从 `memory/<类型>/_template.md` 读取结构（skills 与 department-focus 无模板，参考现有文件）。
5. **填充内容**：按模板结构填充用户提供的情境、决策、逻辑等（若来源为 Word，则使用步骤 3 的总结结果）。已完成事项：每行格式为 `- 任务描述（来源：xxx）`，来源可选，若用户未提供则写「待补充」。
6. **写入文件**：保存到对应路径。已完成事项：若当日 completed-YYYYMMDD.md 已存在，则追加到「事项列表」下；否则按模板新建。若该记忆同时关联任务（如决策、部门重点、已完成），需在对应任务相关记忆模块中保持一致性（如决策的后续待办、部门重点中的任务项）。

## Word 文档（含图片）处理流程

当需要更新的记忆来源于 Word 文档（.docx），且文档中包含图片时，**必须同时读取文字与图片内容，总结提炼后再写入记忆模块与任务模块**。处理时请结合 **docx 技能**（文本提取、解包与 media 访问）进行。

### 适用条件

- 用户明确提供或引用了一个 `.docx` 文件作为记忆/任务来源；
- 或用户说「把这份 Word 记到记忆里」「根据这个 docx 记录决策」等，并附带或指向该文档。

### 处理步骤

1. **提取正文文字**
   - 使用 pandoc 将 .docx 转为 Markdown，保留结构（若需保留修订信息可用 `--track-changes=all`）：
     ```bash
     pandoc --track-changes=all path-to-file.docx -o temp.md
     ```
   - 或仅要纯文本时可直接读取 `temp.md` 作为正文基础。

2. **判断是否含图片并提取图片内容**
   - 解包文档：`python ooxml/scripts/unpack.py <file.docx> <unpack_dir>`（若环境无此脚本，可改用能解压 .docx 的工具查看 `word/media/`）。
   - 检查 `word/media/` 下是否存在图片文件（如 .png、.jpg、.emf 等）。
   - 若存在图片：
     - **可读格式**（.png、.jpg 等）：直接读取图片文件，用多模态能力理解每张图的内容（图表、截图、示意图等），并生成简短文字描述（如「图：xxx 流程图」「图：某数据表格」）。
     - **仅矢量/不易直接解读的格式**（如 .emf）：可先按 docx 技能将文档转为 PDF 再转为页面图片，对关键页或含图页面做视觉理解并总结；或在文档中标注「含示意图/图表，见原文」并在总结中说明位置与作用。
   - 将「正文文字」与「每张图片的文字描述」按在文档中的逻辑顺序整理成一份「正文 + 图注/图意」的完整内容草稿。

3. **总结提炼**
   - 基于上述「正文 + 图意」草稿，按本次要写入的**记忆类型**做提炼：
     - 决策：提炼情境、选项、结论、逻辑与后续行动；
     - 成功/失败/冲突：提炼情境、关键行动与原因/结果；
     - 部门重点/已完成：提炼任务项、责任与时间节点；
     - 技能栈：提炼与技能相关的要点。
   - 总结中若有关键信息来自某张图（如指标、流程），需在记忆文本中明确写出，不写「见附图」而写清结论或数据要点。

4. **写入记忆与任务模块**
   - 将**提炼后的总结**按「执行流程」步骤 4～6 写入对应记忆文件（如 `memory/decisions/decision-NNN.md`、`memory/completed-tasks/completed-YYYYMMDD.md`、`memory/department-focus/focus-YYYYMMDD.md` 等）。
   - 若该条记忆同时对应「任务」维度（例如既是决策又产生待办），需在相关任务模块中一致体现（例如决策中的「后续行动」与部门重点或已完成中的任务项对应），避免同一事实在记忆与任务中表述矛盾。

### 注意事项

- 不单独把 Word 或图片原文件路径写入记忆；只写入**提炼后的文字总结**，必要时在总结中注明「依据附件/Word 文档整理」。
- 若 Word 无图片，仅做文字提取与总结即可，无需执行图片相关步骤。
- 依赖与工具：文本提取可用 pandoc；解包与 media 访问参考 **docx 技能**中的 Raw XML access、Unpacking、word/media 说明；图片理解使用当前模型的多模态读图能力。

## 模板参考

- 决策：情境描述、决策结果、决策逻辑、后续反思
- 成功：情境、决策与行动、成功因素
- 失败：情境、决策与行动、失败原因
- 冲突：冲突背景、处理方式、结果与反思

## 注意事项

- 新建文件时，frontmatter 中 id 与文件名一致（如 decision-005）。
- date 使用 YYYY-MM-DD 格式，默认当日。
- 技能栈为单文件，在合适分类下追加或更新条目。
- 已完成事项：写入 `memory/completed-tasks/completed-YYYYMMDD.md`，当日文件已存在则追加；任务描述与来源将用于今日待办查询时的过滤匹配。
- **Word 文档（含图片）**：当记忆来源为 .docx 且文档中含图片时，必须先按「Word 文档（含图片）处理流程」提取文字与图片、总结提炼，再写入记忆与任务模块；不得仅写入「见附件」或原文路径。
